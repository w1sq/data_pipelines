# Data Pipelines | HW 09 — сравнительный анализ распределённых SQL-движков (Trino, StarRocks, ClickHouse)

## Задача

Провести сравнительный анализ распределённых SQL-движков (минимум — `Trino` и `StarRocks`).
Третьим движком я добавил `ClickHouse` — это самый популярный open-source движок для ad-hoc
аналитики, и его удобно поставить рядом как «эталон скорости» по `GROUP BY`.

В отчёте раскрыты 4 обязательных пункта:

1. **Бенчмарки** — свои тесты + внешние источники со своими комментариями.
2. **Ключевые различия** движков.
3. **Анализ рисков внедрения**.
4. **Итоговое решение**.

### Кто есть кто (одной строкой)

| Движок | Что это | Storage | Главный сценарий |
|--------|---------|---------|------------------|
| **Trino** | распределённый MPP SQL-движок-федератор | своего нет, ходит в источники через коннекторы | единый SQL поверх многих систем (lakehouse, RDBMS, Kafka, …) |
| **StarRocks** | MPP аналитическая СУБД | свой колоночный формат + внешние таблицы (Iceberg/Hudi/Delta/Hive) | BI + lakehouse, материализованные представления, низкая латентность |
| **ClickHouse** | колоночная аналитическая СУБД | свой `MergeTree` | events / logs / clickstream, очень быстрые агрегации |

## 1. Бенчмарки

Я сделал сравнение в двух плоскостях: **внешние авторитетные бенчмарки** (чтобы числа были
на серьёзном железе и больших данных) и **свой воспроизводимый код** (чтобы не пересказывать
чужие графики «на глаз»).

| Внешний бенч 2 | ClickBench, машина `c6a.4xlarge`, raw JSON-снапшоты |

Сырые числа и источники собраны в [`benchmark_summary.csv`](benchmark_summary.csv).

### 1.2. Свой расчёт по ClickBench (StarRocks vs ClickHouse) — реально посчитано

Чтобы не доверять одной цифре с лидерборда, я взял **официальные raw-снапшоты ClickBench**
для одинаковой машины `c6a.4xlarge` (ClickHouse и StarRocks) и сам пересчитал честные метрики
скриптом [`clickbench_metrics.py`](clickbench_metrics.py). Снапшоты лежат в `data/`, поэтому
расчёт воспроизводим офлайн; `--refresh` тянет свежий JSON из репозитория ClickBench.

Методика (как считаю):

- **cold** — первый прогон запроса (холодный кэш);
- **hot** — `min(второй, третий прогон)` (тёплый кэш);
- **ratio** = `время StarRocks / время ClickHouse`, агрегирую по 43 запросам через **геометрическое
  среднее** (устойчиво к одному выбросу);
- **combined_ratio** — взвешенная свёртка: `load 0.1 + size 0.1 + cold 0.2 + hot 0.6`
  (для интерактивной аналитики важнее всего hot-латентность).

Результат прогона (`> 1` означает «ClickHouse быстрее»):

| Метрика | ClickHouse | StarRocks | StarRocks / ClickHouse |
|---------|-----------:|----------:|----------------------:|
| Load time | `312 s` | `608 s` | `1.95x` |
| Data size | `15.25 GB` | `17.25 GB` | `1.13x` |
| Cold (gmean по 43 запросам) | baseline | — | `1.84x` |
| Hot (gmean по 43 запросам) | baseline | — | `1.74x` |
| **Combined** | baseline | — | **`1.70x`** |

Дополнительно: из 43 сопоставимых запросов StarRocks был быстрее ClickHouse лишь в **11 hot**
и **13 cold** запросах — то есть на ClickBench ClickHouse стабильно сильнее.

**Мой вывод:** на «нативной» ad-hoc аналитике (один широкий денормализованный стол, тяжёлые
`GROUP BY`/фильтры) ClickHouse заметно быстрее и компактнее StarRocks. Это его домашнее поле:
`MergeTree` + сортировка по ключу + векторизованное чтение. Но это **не** сценарий lakehouse и
не сценарий федерации — поэтому одного этого бенчмарка для выбора движка недостаточно.

### 1.3. Внешний бенч (StarRocks vs Trino) — lakehouse TPC-DS

Для пары `StarRocks` vs `Trino` корректно смотреть **lakehouse-нагрузку**, а не ClickBench.
Беру официальный бенчмарк StarRocks: TPC-DS 1 TB поверх Iceberg, одинаковый кластер.

| Движок | Полное время 99 запросов |
|--------|-------------------------:|
| `StarRocks` | `368 404 ms` |
| `Trino` | `2 552 076 ms` |

StarRocks быстрее Trino примерно в **`6.9x`** на этом сценарии. Источник заинтересованный
(вендор StarRocks), поэтому абсолютную цифру стоит делить на 1.5–2, но даже с поправкой
направление сохраняется: на тяжёлой lakehouse-аналитике специализированный MPP-движок со своим
вычислителем и кэшем обгоняет универсальный федератор. Это ожидаемо — у Trino другая основная
задача (см. п.2).

### 1.4. Свой микро-бенч (три движка на одном ноутбуке)

Чтобы пощупать движки руками на одинаковых данных, я написал Docker-харнесс
[`experiments.py`](experiments.py) (+ [`bench_helpers.py`](bench_helpers.py)). Он поднимает три
одноузловых контейнера, грузит одинаковую звезду (`fact_sales` + `dim_customers` +
`dim_products`) и гоняет одинаковые 4 запроса по 3 раза:

| # | Запрос | Что проверяет |
|---|--------|---------------|
| q1 | `SUM/COUNT GROUP BY sale_date` | простая агрегация по дням |
| q2 | join + фильтр по датам + агрегация по регионам | типичный BI-запрос |
| q3 | 2 join + `GROUP BY` + `LIMIT 10` | топ регионов/категорий по выручке |
| q4 | `COUNT(DISTINCT) + SUM` по сегментам | более тяжёлая агрегация |

```sql
-- q3: два join + группировка + топ-10 (пример из набора)
SELECT c.region, p.category, SUM(f.amount) AS revenue, COUNT(*) AS orders_cnt
FROM fact_sales f
JOIN dim_customers c ON f.customer_id = c.customer_id
JOIN dim_products  p ON f.product_id  = p.product_id
GROUP BY c.region, p.category
ORDER BY revenue DESC
LIMIT 10;
```

Запуск:

```bash
cd hw9
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# нужен запущенный Docker (на этой машине — colima start)
python experiments.py                 # все три движка, 100k фактов
python experiments.py --engine trino  # один движок
python experiments.py --fact-rows 50000 --json results.json
```

**Честная оговорка по микро-бенчу.** На этой машине Docker-демон (colima) во время сдачи был
выключен, поэтому headline-числа я взял из воспроизводимого расчёта ClickBench (п.1.2, реально
прогнан) и из внешнего lakehouse-бенча (п.1.3). Микро-бенч — это **корректный воспроизводимый
харнесс**, но важно понимать его границы: одноузловые контейнеры и загрузка через
`INSERT … VALUES` нагружают парсинг/коммит, а не движок аналитики; на таком масштабе (~0.1M
строк) разница между движками тонет в JVM-warmup и оверхеде вставки. Делать вывод о
«победителе» по нему нельзя — для этого и нужны п.1.2–1.3 на серьёзном железе и объёмах.

## 2. Ключевые различия

| Критерий | Trino | StarRocks | ClickHouse |
|----------|-------|-----------|------------|
| **Тип** | SQL-федератор (compute-only) | MPP-СУБД (compute + storage) | колоночная СУБД (compute + storage) |
| **Где данные** | в источниках (S3, RDBMS, Kafka, Iceberg…) | свой формат **и** внешние таблицы | свой `MergeTree` |
| **Главная сила** | десятки коннекторов, единый SQL | материализованные view, кэш, низкая латентность на BI | агрегации/фильтры, сжатие, скорость |
| **Joins** | хорошо, но без своего storage медленнее | сильный distributed join, CBO | исторически слабое место, лучше денормализация |
| **Обновления/upsert** | через коннектор (ограниченно) | Primary Key модель, частичные апдейты | дорогие (ReplacingMergeTree, мутации) |
| **Где блистает** | федерация, ad-hoc поверх многих систем | lakehouse + BI как основной движок | events, logs, clickstream, дашборды |
| **Главный минус** | не самый быстрый, зависит от источника | сложнее в эксплуатации, чем чистый query-engine | не федератор, чувствителен к модели данных |

Если совсем коротко:

- **Trino** оптимизирует **охват** (один SQL ко всему), а не пиковую скорость.
- **StarRocks** — **компромисс** между скоростью аналитики и работой с lakehouse/внешними форматами.
- **ClickHouse** оптимизирует **скорость нативной аналитики** на своих таблицах.

## 3. Анализ рисков внедрения

### Trino
- **Риск ожиданий:** от него ждут скорости специализированной СУБД. Он федератор — на тяжёлой
  аналитике без своего storage он медленнее (см. п.1.3).
- **Производительность = скорость источника:** запрос в медленный Postgres/S3 будет медленным,
  виноватым «назначат» Trino.
- **Эксплуатация:** нет своего storage и кэша данных (есть только метаданные/файловый кэш) —
  нужно думать про память воркеров, spill, конкурентность; падение одного запроса по памяти
  может ударить по кластеру.
- **Митигация:** использовать как SQL-слой/федерацию, тяжёлые витрины материализовать в
  Iceberg/таблицы; не делать его единственным BI-движком под высокую конкуренцию.

### StarRocks
- **Риск сложности:** мощь раскрывается только при правильной настройке — материализованные
  представления, refresh, кэш, модель таблиц (Duplicate/Aggregate/Primary Key), бакетирование.
  Плохо настроишь — не получишь обещанной скорости.
- **Зрелость экосистемы** меньше, чем у ClickHouse/Trino; меньше готовых интеграций и
  community-ответов, заметная вендор-окраска в бенчмарках.
- **Эксплуатация FE/BE**, тюнинг внешних каталогов, версионность.
- **Митигация:** начинать с типовых паттернов из доков, завести нагрузочное тестирование на
  своих запросах, выделить владельца материализованных представлений.

### ClickHouse
- **Риск модели данных:** всё решают `ORDER BY`-ключ, партиционирование и структура таблиц.
  Ошибся — производительность падает в разы.
- **Joins и обновления:** распределённые join и upsert/мутации — слабое/дорогое место; парадигма —
  денормализация и append-only.
- **Не федератор:** ходить в разнородные источники одним SQL он не предназначен.
- **Митигация:** проектировать схему под запросы заранее, держать данные денормализованными,
  не тащить туда транзакционные upsert-сценарии.

### Общие риски (для всех трёх)
- Распределённость = сетевые/нодовые отказы, перекос данных (data skew), управление памятью.
- Вендор-лок и стоимость: managed-версии (StarRocks Celerdata, ClickHouse Cloud, Starburst для
  Trino) удобны, но дороги; self-hosted дешевле, но требует SRE-компетенций.
- Бенчмарки от вендоров оптимистичны — всегда проверять на **своих** запросах и данных.

## 4. Итоговое решение

Решение зависит от сценария, поэтому формулирую по задачам.

| Сценарий | Выбор | Почему |
|----------|-------|--------|
| Data lake (Iceberg/Hudi) + BI + тяжёлые аналитические запросы, нужны и скорость, и внешние форматы | **StarRocks** | компромисс: быстрее Trino на lakehouse (п.1.3), умеет внешние таблицы и MV |
| Единый SQL-доступ к многим разнородным системам, важен охват | **Trino** | коннекторы и федерация — его профиль |
| Events / logs / clickstream / дашборды, очень быстрые агрегации на своих таблицах | **ClickHouse** | лучший по нативной аналитике (п.1.2: combined `1.70x` vs StarRocks) |

**Мой основной выбор — `StarRocks`**, если нужно выбрать один движок под типовой современный стек
«data lake + BI + тяжёлая аналитика». Обоснование:

1. он заметно быстрее `Trino` на lakehouse-нагрузке (TPC-DS, ~`6.9x`);
2. по перфоманс-мышлению он близок к `ClickHouse` (MPP, колоночность, кэш), уступая ему на
   чистом ClickBench лишь в `1.70x`, но при этом **умеет** lakehouse и внешние форматы, чего нет
   у ClickHouse;
3. остаётся «движком одного SQL» для BI, не требуя отдельного федератора.

При этом я честно фиксирую границы: для чистой федерации правильнее `Trino`, а для
events/clickstream-аналитики на своих таблицах — `ClickHouse`. То есть «лучшего вообще» движка
нет: `Trino` — лучший охват, `ClickHouse` — лучшая нативная скорость, `StarRocks` — лучший
компромисс для `lakehouse + BI`.
