## Задача: обработка бинарного потока биржевых сделок

У нас есть низкоуровневый поток с биржи. Каждое сообщение — это **бинарный пакет**
(а не строка/JSON/Parquet), который нужно разобрать своим парсером. После разбора
получается полноценный Python-объект со своими методами и вложенными структурами:

```python
class TradeMessage:
    def __init__(self, symbol, price, volume, flags):
        self.symbol = symbol
        self.price = price
        self.volume = volume
        self.flags = flags          # битовая маска: тип сделки, площадка, аукцион ...

    def is_valid(self) -> bool:
        # сложная доменная валидация: проверка флагов, кросс-площадочные правила,
        # внутренняя согласованность пакета
        ...

    def compute_risk_metrics(self) -> "RiskEnrichedTrade":
        # тяжёлая бизнес-логика: модель риска, обращение к справочникам,
        # построение вложенного объекта с методами
        ...
```

Пайплайн целиком работает на уровне **объектов**, а не таблиц:

```python
(rdd
    .map(parse_binary_packet)      # bytes -> TradeMessage
    .filter(is_valid_trade)        # доменная валидация (метод объекта)
    .map(compute_risk_metrics))    # TradeMessage -> RiskEnrichedTrade
```

Результат каждой функции — **живой Python-объект**, а не плоская строка таблицы.

## Почему здесь RDD выигрывает у DataFrame

### 1. Компактнее — меньше кода

**RDD:** три строки выражают всю логику. Парсер и бизнес-правила остаются обычными
Python-функциями/методами — их не нужно адаптировать под движок.

**DataFrame:** пришлось бы:

- распаковать бинарный пакет во множество колонок;
- явно описать схему (`StructType` со всеми вложенными `StructType`/`ArrayType`);
- переписать всю доменную логику через SQL-выражения или обернуть в UDF
  (что лишает DataFrame его главного преимущества — оптимизатора Catalyst, потому что
  UDF для движка — «чёрный ящик»).

Объект `TradeMessage` с методами и вложенными структурами просто **не ложится**
на табличную модель без громоздкой ручной развёртки.

### 2. Быстрее — ручные оптимизации добавляют скорость

- Бинарный парсинг и риск-метрики — это **CPU-bound** работа над объектами.
  В DataFrame она всё равно уехала бы в Python-UDF, где теряется выигрыш от Catalyst
  и Tungsten, но **добавляются** накладные расходы на сериализацию строк
  Row ↔ Python и на формирование колоночного представления.
- На RDD я управляю гранулярностью вручную: `mapPartitions` позволяет один раз
  на партицию открыть тяжёлые ресурсы (декодер протокола, подключение к справочнику
  рисков, модель), а не на каждую запись.
- Нет «штрафа» за построение и поддержку колоночной схемы для данных, которые
  всё равно живут как объекты.

```python
def process_partition(packets):
    decoder = ProtocolDecoder()          # один раз на партицию
    risk_model = RiskModel.load()        # тяжёлый ресурс переиспользуется
    for raw in packets:
        msg = decoder.parse(raw)
        if msg.is_valid():
            yield msg.compute_risk_metrics()

enriched = rdd.mapPartitions(process_partition)
```

### 3. Надёжнее — возможность управлять памятью

- Я сам выбираю уровень хранения: `persist(StorageLevel.MEMORY_AND_DISK)` или
  `DISK_ONLY` для тяжёлых объектов, чтобы не словить OOM.
- Через `mapPartitions` объекты обрабатываются **потоково** (генератором) — в памяти
  одновременно живёт только текущая запись, а не материализованный батч колонок.
- Я контролирую число партиций (`repartition`/`coalesce`) под реальный размер пакетов
  и доступную память исполнителей. В DataFrame управление памятью во многом скрыто
  внутри движка и подстроено под колоночные данные, а не под произвольные объекты.

### 4. Гибче — то, для чего DataFrame не очень подходит

- Произвольный бинарный формат, кастомный парсер, **вложенные объекты с методами**,
  полиморфизм бизнес-правил — всё это естественно выражается обычным Python-кодом.
- Логика валидации и расчёта риска может быть сколь угодно сложной, ветвиться,
  обращаться к внешним библиотекам — без необходимости укладывать её в реляционную
  модель и SQL-семантику.
- DataFrame заточен под структурированные/полуструктурированные данные с фиксированной
  схемой; здесь же «схема» — это поведение объекта, а не набор колонок.
