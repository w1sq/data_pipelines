# Текст выступления

Сегодня я предложу архитектуру Kafka-пайплайна для обработки документов в «Консультант Плюс». Задача не в том, чтобы просто передавать сообщения между сервисами, а в том, чтобы построить систему, которая надёжно обрабатывает юридические документы, сохраняет порядок изменений, поддерживает историю редакций и при этом масштабируется по нагрузке.

Начну с общей идеи. В Kafka удобно мыслить не очередями, а потоками событий. У нас есть документы разных типов: законы, судебные решения, редакции документов, ссылки между документами и служебные события. Поэтому я разбиваю архитектуру не на один общий поток, а на несколько тематических топиков. Это важно, потому что у каждого типа данных своя интенсивность поступления, своя семантика и свои требования к хранению. Решение строится на базовых сущностях Kafka: топик, партиция, сообщение, ключ, consumer group и журналы хранения.

На входе у меня три сырьевых топика. Первый — для законов и нормативных актов, второй — для судебных решений, третий — для редакций документов. Я сознательно разделяю их на уровне входа, потому что законодательные акты и судебные решения имеют разную природу, разную частоту обновления и разный downstream-профиль. После этого всё сырьё проходит нормализацию, где документы приводятся к единой внутренней схеме. Это даёт единообразие для всех последующих этапов: версии, связи, поиск, аналитика.

Теперь ключевой момент Kafka — партиционирование. В качестве ключа сообщения я использую логический идентификатор документа: для закона это `document_id`, для судебного решения — `case_id`, для редакции — `base_document_id`. Так все события одного документа попадают в одну партицию, и Kafka гарантирует порядок внутри этого документа. Это особенно важно, когда есть последовательность: новая редакция, потом уточнение, потом отмена. Если события одного документа разъедутся по разным партициям, история сломается. Поэтому ключ выбирается так, чтобы сохранять локальный порядок там, где он действительно нужен.

По числу партиций: для самых горячих входных потоков — законов и судебных решений — я делаю 24 партиции, это даёт запас по горизонтальному масштабированию. Для потока редакций, истории версий и связей достаточно 12 партиций: там важнее предсказуемый порядок, чем максимальная распараллеленность. Для DLQ оставляю 4 партиции — это аварийный канал, а не основная нагрузка.

Теперь про хранение. В Kafka есть две базовые политики очистки журнала: удаление и сжатие. Для сырых топиков я использую `delete` с retention около 7 дней — это буфер на случай инцидентов или временной недоступности downstream. Для нормализованного потока ставлю retention подольше, например 30 дней, чтобы была возможность переобработки и аудита.

Для потока текущего состояния документа я использую `compact`: Kafka оставляет последнюю запись по каждому ключу, то есть по каждому документу. Именно это нужно для справочника актуальных документов — нас интересует не вся история в одном месте, а последнее состояние. А для истории редакций использую смешанную политику `compact,delete`: мы сохраняем и актуальность, и часть истории, но контролируем объём. Если нужен полноценный юридический архив на годы, его лучше выносить в Data Lake или использовать tiered storage, а Kafka держать как оперативный слой.

Теперь про доставку. Я не выбираю `at most once`, потому что потеря сообщения в юридическом контуре недопустима. Основной режим — `at least once`: сообщение может прийти повторно, но не должно потеряться. Чтобы это работало надёжно, producer должен быть идемпотентным и отправлять с `acks=all`, репликация топиков — не меньше 3, а `min.insync.replicas` — 2. Тогда даже при падении одного брокера система продолжит работать.

Если есть цепочка Kafka ? Kafka, например нормализация или versioning, я использую транзакции — это даёт более сильную семантику внутри шины. Но даже тогда downstream-сервисы должны быть идемпотентными, потому что при внешних эффектах полностью избавиться от повторов сложно. Поэтому у каждого события есть `event_id`, а для редакций — пара `document_id + version`, и consumer может безопасно проверить, что уже применял это событие.

Отдельно про consumer groups. Я выделяю несколько независимых групп. `cg-normalizer` приводит документы к стандартному виду. `cg-versioning` строит историю редакций и актуальное состояние. `cg-relation-extractor` извлекает связи и цитируемость. `cg-search-indexer` пишет в поисковый индекс. `cg-analytics-sink` выгружает данные в хранилище для аналитики и ML. И `cg-replay-recovery` отдельно разбирает DLQ и запускает повторную обработку. Каждый бизнес-процесс живёт независимо, не блокирует другие и масштабируется своими ресурсами.

Если смотреть на систему целиком: источники публикуют документы в raw-топики, normalizer приводит их к единому формату, versioning строит историю и текущее состояние, relation extractor достаёт ссылки, search indexer обновляет поиск, а analytics sink отправляет данные в DWH. Все ошибки уходят в DLQ, откуда их можно повторно обработать. То есть Kafka здесь — не единая очередь, а центральная шина событий с разделением по доменам, управляемым хранением и отдельными контурами чтения.

В итоге архитектура даёт три главных преимущества. Первое — порядок внутри одного документа сохраняется. Второе — история редакций не теряется. Третье — система масштабируется независимо по каждому потоку. Для юридической платформы это особенно важно, потому что нужны одновременно точность, отказоустойчивость и быстрый поиск. Именно поэтому Kafka хорошо подходит для такого пайплайна.
