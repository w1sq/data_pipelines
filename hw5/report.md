## Как замеряются метрики

1. Для каждого кейса генерируется синтетический датасет (`spark.range` + детерминированные преобразования).
2. Сравниваются **baseline** (наивная реализация с проблемой) и **optimized** (оптимизированная).
3. Замеряется wall-clock (`time.perf_counter`) вокруг материализующего action (`collect`/`count`).
4. Метрики стадий (shuffleRead/Write, spill, executorRunTime, numTasks) подтягиваются через Spark UI
   REST API (`/api/v1/applications/{id}/stages`).

### Две тонкости измерения, которые пришлось решить

**(а) Метрики по дельте стадий.** Наивно REST возвращает суммарные метрики за всё время приложения,
и числа соседних кейсов «протекают» друг в друга. Класс `StageMetrics` (context manager) запоминает
`stageId` **до** запуска и суммирует только **новые** стадии.

**(б) Адрес Spark UI.** `uiWebUrl()` отдаёт LAN-IP (`http://172.x.x.x:4040`), соединение к которому
сбрасывается (`Connection reset by peer`). Пришлось брать из него только порт и ходить на `localhost` —
иначе все метрики были нулевыми.

```92:103:hw5/experiments.py
def _ui_url(spark: SparkSession) -> Optional[str]:
    # uiWebUrl() часто возвращает LAN-IP (например http://172.x.x.x:4040), к которому
    # соединение сбрасывается. Берём только порт и обращаемся к localhost.
    port = 4040
    try:
        url = spark.sparkContext._jsc.sc().uiWebUrl().get()
        if url and ":" in url:
            port = int(url.rsplit(":", 1)[1])
    except Exception:
        pass
    return f"http://localhost:{port}"
```

## Эксперимент 1 — Broadcast join vs Shuffle join

**Проблема.** `orders (4M) JOIN countries (200)` по умолчанию идёт через sort-merge join: обе таблицы
шафлятся по `country_id`. Гонять `orders` по сети ради крошечного справочника — расточительство.

**Оптимизация.** `broadcast(countries)` рассылает справочник на все executors, join выполняется
локально (map-side), shuffle большой таблицы исчезает.

| Кейс | time_s | executorRunTime | speedup |
|------|--------|-----------------|---------|
| Baseline (shuffle join) | 1.404 | 2863 ms | — |
| **Broadcast join** | **0.087** | 63 ms | **?16.1** |

**Вывод.** Для join с маленьким справочником broadcast — почти бесплатная оптимизация (executorRunTime
упал в ~45 раз). Граница: broadcast большой таблицы приведёт к OOM на драйвере/executors.

## Эксперимент 2 — Skew & Salting

**Проблема.** 90% строк имеют ключ `HOT_KEY`. При `groupBy` они попадают в одну партицию — один
task-straggler обрабатывает почти весь объём, остальные простаивают.

**Оптимизация.** Добавляем `salt` (0..15) к ключу, делаем частичную агрегацию по `key+salt`, затем
сводим частичные результаты по исходному `key`.

| Кейс | time_s | executorRunTime | speedup |
|------|--------|-----------------|---------|
| Baseline (`groupBy`) | 0.929 | 4629 ms | — |
| **Salting (K=16)** | **0.652** | 2964 ms | **?1.43** |

**Trade-off.** Salting распараллелил «горячую» партицию (executorRunTime ?36%), wall-clock ?30%.
Выигрыш умеренный из-за дешёвого `count`; на тяжёлой агрегации/join и большом кластере он значительно выше.

## Эксперимент 3 — repartition / coalesce

**Проблема.** Число партиций = степень параллелизма. Слишком мало -> недогрузка CPU; слишком много ->
overhead планирования, мелкие shuffle-файлы и **spill**.

| Конфигурация | time_s | shuffleWrite | spill (mem) | partitions |
|--------------|--------|--------------|-------------|-----------|
| Baseline | 0.569 | 51 KB | 0 | 8 |
| repartition(2) | 2.317 | 568 KB | 0 | 2 |
| repartition(8) | 1.398 | 663 KB | 0 | 8 |
| repartition(16) | 0.768 | 788 KB | 0 | 16 |
| repartition(50) | 1.014 | 1.3 MB | 0 | 50 |
| repartition(200) | 2.556 | 3.6 MB | 0 | 200 |
| repartition(500) | 6.247 | 7.7 MB | 54 MB | 500 |
| repartition(1000) | 10.721 | 15.5 MB | **218 MB** | 1000 |
| coalesce(2) | 0.338 | 21 KB | 0 | 2 |
| **coalesce(4)** | **0.261** | 31 KB | 0 | 4 |

**Вывод.** Кривая U-образная: и недостаток (2 партиции, 2.3 с), и избыток (1000 партиций, 10.7 с)
бьют по wall-clock — разница **?40**. Видно, как при 500+ партициях начинается spill в память
(до 218 МБ), потому что лишний `repartition` сам по себе вызывает дорогой shuffle. `coalesce` дешевле:
он уменьшает число партиций без полного shuffle и даёт лучший результат (0.26 с).

## Бонусные эксперименты

### Эксперимент 4\* — Python UDF vs нативные функции

**Проблема.** Python UDF гоняет каждую строку через сериализацию JVM<->Python, и Catalyst видит её как
«чёрный ящик» (нет codegen).

**Оптимизация.** Та же логика на `F.when(...).otherwise(...)` остаётся в JVM и компилируется (codegen).

| Кейс | time_s | executorRunTime | speedup |
|------|--------|-----------------|---------|
| Python UDF | 3.203 | 23837 ms | — |
| **native when/otherwise** | **0.475** | 1882 ms | **?6.75** |

**Вывод.** На 8M строк нативный вариант быстрее в **6.75?**, а суммарное процессорное время (executorRunTime)
меньше в **12.7?**. Главное практическое правило: не пишите UDF, если есть нативный аналог.

### Эксперимент 5\* — cache() vs пересчёт DAG (отрицательный результат)

**Проблема.** Дорогой DataFrame переиспользуется в 3 action и пересчитывается с нуля каждый раз.

**Оптимизация (ожидаемая).** `cache()` + один материализующий `count()`.

| Кейс | time_s | inputBytes | speedup |
|------|--------|-----------|---------|
| no cache (3? пересчёт) | 0.940 | 0 | — |
| cache + reuse | 1.990 | 505 MB | **?0.47** |

**Честный вывод: на локальной машине cache оказался в 2 раза медленнее.** Причины:
(1) преобразования (`sin/sqrt/abs`) дёшевы, пересчёт почти бесплатен; (2) материализация кэша добавляет
обязательный лишний проход (`count`) и сериализацию ~505 МБ в память (виден `inputBytes` чтения из кэша).
`cache()` окупается, когда **исходные данные дорого читать** (большой shuffle/join/чтение с диска) и
переиспользований много. Это важная иллюстрация: оптимизация без понимания контекста вредит.

### Эксперимент 6\* — AQE против skew без ручного salting (отрицательный результат)

**Проблема.** Skew join требует ручного salting — кода, который надо писать и поддерживать. Хочется,
чтобы Spark разруливал skew сам.

**Оптимизация.** Adaptive Query Execution (`adaptive.enabled=true`, `skewJoin.enabled=true`) дробит
перекошенные партиции в рантайме; `autoBroadcastJoinThreshold=-1` форсит sort-merge join.

| Кейс | time_s | spill (mem) | spill (disk) | speedup |
|------|--------|-------------|--------------|---------|
| AQE off | 2.863 | 0 | 0 | — |
| AQE on (skewJoin) | 3.676 | 157 MB | 1.2 MB | **?0.78** |

**Честный вывод: на маленьких локальных данных AQE дал замедление.** Понизив пороги, мы заставили
skewJoin дробить партиции агрессивно — он наплодил мелких задач и вызвал spill 157 МБ. AQE — про
**большие данные на кластере**, где автодробление перекоса реально экономит; на игрушечном датасете
его накладные расходы перевешивают. Тоже поучительно: «современная» фича не панацея.

### Эксперимент 7\* — column pruning + predicate pushdown (Parquet)

**Проблема.** `read.parquet(...).select("*").filter(...)` читает с диска все колонки, включая тяжёлую
`payload` (SHA-256), и фильтрует уже в памяти.

**Оптимизация.** `select("country_id","amount").filter(...)` — Spark передаёт в Parquet-reader список
колонок (column pruning) и предикат (predicate / row-group pushdown).

| Кейс | time_s | executorRunTime | speedup |
|------|--------|-----------------|---------|
| read all cols | 0.611 | 3248 ms | — |
| **pruned + pushdown** | **0.206** | 947 ms | **?2.96** |

**Вывод.** Чтение только нужных колонок ускорило запрос в ~3? (CPU ?70%). Главное преимущество
колоночного формата на практике; эффект растёт с числом и «весом» отбрасываемых колонок.

## Выводы

1. **Broadcast** — почти бесплатное ускорение для join с маленьким справочником (?16).
2. **Salting** — лечит перекос данных ценой роста shuffle; выигрыш растёт с размером кластера.
3. **Партиционирование** — кривая U-образная (?40 между крайностями); избыток партиций вызывает spill;
   `coalesce` дешевле `repartition` для уменьшения.
4. **UDF != native** — нативные функции быстрее Python UDF (?6.75, CPU ?12.7).
5. **cache()** — окупается только при дорогом чтении исходника и многих переиспользованиях; иначе вредит
   (здесь ?0.47).
6. **AQE** — выигрывает на больших данных; на маленьком локальном датасете накладные расходы перевесили
   (здесь ?0.78).
7. **Pruning + pushdown** — читаем меньше данных c диска; работа делегируется формату Parquet (?3).

**Главный вывод:** у каждой оптимизации есть цена. Самое интересное в этих экспериментах — два
**отрицательных** результата (cache, AQE): они показывают, что «оптимизацию» нельзя применять вслепую,
нужно смотреть на профиль данных, размер кластера и форму запроса.
